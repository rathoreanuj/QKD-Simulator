\begin{abstract}
Quantum Key Distribution (QKD) enables key establishment with information-theoretic security guarantees, but experimental learning and early-stage benchmarking remain constrained by hardware availability and fragmented tooling. This paper presents a unified software QKD simulator supporting four canonical protocols (BB84, B92, E91, BBM92) with realistic channel and detector modeling. The implementation integrates a Python/Qiskit simulation core with both a desktop interface (Tkinter + Matplotlib) and a web architecture (React + Node/Express + Python bridge), enabling local and remote experimentation with a shared backend. To strengthen statistical reliability, the experimental section reports repeated-run results using mean, standard deviation, and 95\% confidence intervals. Across 20 baseline runs (10,000 qubits/run), BB84 achieved the highest mean key-rate estimate (160,045.15 Hz), while E91 provided Bell-correlation diagnostics with mean S = 2.12. Under E91 eavesdropping mode, mean S dropped to 1.58, indicating reduced nonclassical correlation evidence. The platform is suitable for QKD education, protocol-level comparative analysis, and pre-deployment design-space studies. We also provide camera-ready reporting guidance and a reproducibility template aligned with reviewer expectations for IEEE conference submissions.
\end{abstract}

\section{Introduction}
Quantum-safe communication is increasingly relevant as quantum computing threatens hardness assumptions used by classical public-key systems. QKD mitigates this risk by deriving security from quantum mechanical principles rather than computational complexity. However, practical access to QKD hardware is limited, and many learners encounter either theoretical-only materials or software tools that are protocol-specific and difficult to compare under consistent conditions.

This work presents a unified QKD simulator designed for education, rapid security experimentation, and pre-deployment sensitivity analysis. It combines prepare-and-measure protocols (BB84, B92) and entanglement-based protocols (E91, BBM92) in one simulation core, with realistic impairment toggles and two user-facing modes: desktop GUI and web application.

The manuscript is written as an implementation-focused IEEE conference paper: architecture, mathematical model, reproducible methodology, repeated-run statistics, and camera-ready reporting strategy are all included to support submission readiness.

\section{Problem Statement and Objectives}
QKD practitioners at beginner and intermediate levels often face four constraints:
\begin{enumerate}
\item Limited access to physical photonic/QKD testbeds.
\item Inability to compare multiple protocols within one consistent environment.
\item Weak support for realistic channel/device impairments in educational tools.
\item Inconsistent reporting practices for stochastic simulation outputs.
\end{enumerate}

Accordingly, this project targets the following objectives:
\begin{enumerate}
\item Build one simulator for BB84, B92, E91, and BBM92.
\item Model source/fiber/detector losses, SOP uncertainty, perturbations, and eavesdropping.
\item Expose QBER and (for E91) CHSH-style S-statistic security signals.
\item Provide desktop and web interfaces over a common simulation backend.
\item Report results with repeated-run statistics (mean/std/CI) for stronger reproducibility.
\end{enumerate}

\section{System Architecture}
\subsection{Layered Design}
The platform is structured as three layers:
\begin{enumerate} 
\item \textbf{Simulation Core (Python + Qiskit)}: circuit generation, impairment injection, sifting, and metric computation.
\item \textbf{Desktop Layer (Tkinter + Matplotlib)}: single-run and multi-run GUI workflows with progress tracking and per-protocol cards.
\item \textbf{Web Layer (React + Node/Express + Python bridge)}: API-driven browser workflow for all-protocol simulation and parameter sweeps.
\end{enumerate}

\subsection{Execution and Deployment Notes}
The web backend exposes health, single-protocol, all-protocol, and sweep endpoints. The project includes containerization support (frontend build + backend runtime + Python dependencies) for reproducible deployment. The simulator uses Qiskit Aer when available and falls back to BasicProvider otherwise.

\section{Mathematical Modeling}
\subsection{Combined Efficiency and Loss}
The combined efficiency model is:
\begin{equation}
\eta_{tot} = \eta_s \cdot 10^{-\frac{dL}{10}} \cdot \eta_d
\label{eq:etatot}
\end{equation}
where $\eta_s$ is source efficiency, $d$ is distance (km), $L$ is attenuation (dB/km), and $\eta_d$ is detector efficiency.

Loss probability is:
\begin{equation}
P_{loss} = 1 - \eta_{tot}
\label{eq:ploss}
\end{equation}

\subsection{QBER and Key-Rate Approximation}
QBER is estimated from a disclosed check subset:
\begin{equation}
QBER = \frac{N_{error}}{N_{checked}} \times 100\%
\label{eq:qber}
\end{equation}

Protocol-dependent key-rate estimate:
\begin{equation}
R_{key} \approx R_s \cdot \eta_{tot} \cdot (1-f_{qber}) \cdot \kappa_p
\label{eq:keyrate}
\end{equation}
where $R_s$ is source rate and $\kappa_p$ captures protocol-specific sifting behavior.

\subsection{E91 Bell-Correlation Diagnostic}
The CHSH-style statistic is computed as:
\begin{equation}
S = |E(a_1,b_1) - E(a_1,b_2) + E(a_2,b_1) + E(a_2,b_2)|
\label{eq:chsh}
\end{equation}
with $S>2$ indicating Bell-inequality violation in idealized interpretation.

\subsection{Statistical Protocol}
To reduce one-off stochastic bias, repeated runs are used. For each metric $x$ with sample size $n$:
\begin{equation}
\mu = \frac{1}{n}\sum_{i=1}^{n}x_i, \quad
\sigma = \sqrt{\frac{1}{n-1}\sum_{i=1}^{n}(x_i-\mu)^2}
\label{eq:meanstd}
\end{equation}

95\% confidence interval (normal approximation):
\begin{equation}
CI_{95\%} = \mu \pm 1.96\frac{\sigma}{\sqrt{n}}
\label{eq:ci}
\end{equation}

\subsection{Parameter Set Used for Reported Multi-Run Statistics}
The baseline repeated-run study uses 10,000 qubits per run and the following parameters: source rate = 100 MHz, source efficiency = 0.05, fiber length = 25 km, fiber loss = 0.3 dB/km, detector efficiency = 0.2, perturb probability = 0.02, SOP deviation = 0.1 rad, QBER check fraction = 0.1, with losses/perturbation/SOP enabled.

\section{Discussion}
\subsection{Interpretation of Statistical Findings}
The repeated-run framework clarifies which outputs are deterministic under fixed inputs (key-rate model in the current implementation) and which outputs are stochastic (key length and S-statistic due to finite sampling and basis matching). This distinction is important for reviewer interpretation and for preventing overclaims from single-run snapshots.

\subsection{Security Interpretation}
In the reported E91 eavesdropping configuration, S-statistic degradation is a clearer signal than QBER. This does not invalidate QBER-based analysis; rather, it emphasizes that protocol-specific security indicators should be interpreted jointly and in the context of implementation choices.

\subsection{Threats to Validity}
\begin{itemize}
\item Finite-run statistics at low effective efficiency can produce sparse-key artifacts.
\item Current attacker model is simplified and does not cover advanced coherent/adaptive attacks.
\item Finite-key post-processing (error correction/privacy amplification leakage accounting) is not yet integrated.
\end{itemize}

\section{Reproducibility and Reporting Checklist}
\begin{enumerate}
\item Report full parameter dictionary and enabled toggles per experiment.
\item State number of runs and qubits per run for each table.
\item Report mean, SD, and confidence intervals (not only point values).
\item Separate deterministic model outputs from stochastic outputs during interpretation.
\item Archive scripts/configuration and generated plots with version/date metadata.
\end{enumerate}

\section{Statistical Appendix: Multi-Run Script and Seed-Control Protocol}
\subsection{Run-Script Pseudocode}
The repeated-run statistics routine follows these steps:
\begin{enumerate}
\item Initialize a base parameter dictionary, protocol list, and run count.
\item For each protocol, execute $n$ runs and store sampled metrics (key rate, QBER, key length, S-statistic when available, and combined efficiency).
\item Compute sample mean and sample standard deviation for each metric.
\item Compute 95\% confidence intervals as $\mu \pm 1.96\sigma/\sqrt{n}$.
\item Repeat with eavesdropping enabled for the E91 adversarial case.
\item Repeat across selected sweep points (e.g., fiber length) for sensitivity reporting.
\end{enumerate}

\subsection{Seed-Control Protocol}
To improve reproducibility while preserving stochastic analysis integrity, use a two-level seed policy:
\begin{enumerate}
\item \textbf{Experiment seed:} Fix a master seed $S_0$ per experiment batch and store it in the result metadata.
\item \textbf{Run seed derivation:} For run index $i$, derive deterministic sub-seeds (e.g., $S_i = S_0 + i$) for Python random and NumPy random generators.
\item \textbf{Library-level seeding:} If backend simulators expose explicit seed controls, set them from the same $S_i$ to align circuit-level randomness.
\item \textbf{Audit trail:} Log $S_0$, list of $S_i$, commit hash, configuration dictionary, and script version in a run manifest file.
\item \textbf{Cross-check mode:} Re-run at least one batch with identical seeds to confirm exact repeatability of deterministic paths.
\end{enumerate}

Recommended seed-control snippet (implementation hint): initialize \texttt{random.seed(S\_i)} and \texttt{np.random.seed(S\_i)} at the start of each run loop, then call the protocol simulation routine.

\section{Conclusion}
This paper presents an implementation-centered, multi-protocol QKD simulator that combines practical impairment modeling with dual desktop-web accessibility. The revised experimental section adopts repeated-run statistics and confidence intervals to improve scientific rigor and reviewer confidence. Results confirm expected protocol-wise key-rate ordering under the implemented model, and demonstrate that E91 Bell-correlation diagnostics provide meaningful additional security information in adversarial settings. The platform is directly useful for QKD teaching, protocol comparison, and early-stage system design studies.
